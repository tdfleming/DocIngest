# Milestone v1.0.1: Graph RAG Extension

**Status:** ✅ SHIPPED 2026-04-17
**Extends:** v1.0 MVP (shipped 2026-03-04)
**Phases:** 8-15
**Total Plans:** 12

## Overview

Layered a knowledge-graph pipeline onto the v1.0 MVP — spaCy-based entity/relationship extraction over chunked content, MongoDB graph store with tenant isolation, on-demand multi-resolution Leiden community detection with TF-IDF extractive summaries, and gap-closure work to finish audit items on traceability, lifecycle cleanup, API surface, and code-quality debt. All functionality is gated by `GRAPH_RAG_ENABLED`, so the v1.0 MVP remains operable without any graph features.

Phases 8-11 (Graph RAG feature build) landed before the formal milestone audit surfaced gaps. Phases 12-15 closed every audit gap: 4 plans in phase 12 (pure docs), 1 plan in phase 13 (lifecycle wiring), 1 plan in phase 14 (API surface), 1 plan in phase 15 (tech-debt cleanup). Each phase has a VERIFICATION.md report confirming must-haves satisfied.

## Milestone Summary

### Key Decisions

**Feature gating — `GRAPH_RAG_ENABLED` master switch** (Phase 8): Every graph pipeline stage (API routes, worker, chunker enqueue, lifespan, frontmatter status) gated at call sites. Outcome: MVP remains fully operable without Graph RAG.

**Pydantic v2 + StrEnum pattern** (Phase 8): Followed the exact `document.py` model pattern for graph models. Outcome: Consistent codebase, zero model migration required.

**spaCy lazy-load singleton with threading.Lock** (Phase 9): Mirrored `embedding.py` pattern. Outcome: `en_core_web_lg` (~500MB) loads exactly once per process, only on first use.

**Worker version-based self-cleanup** (Phase 10): Graph worker clears stale data when `version > 1` || `graph_status is not None`. Outcome: Race safety maintained even after phase 13 added synchronous route cleanup.

**CPMVertexPartition for resolution-parameterized Leiden** (Phase 11): Chosen over ModularityVertexPartition for per-level resolution control. Outcome: Hierarchical communities at multiple granularities.

**Retroactive scoping of Graph RAG into v1.0** (Phase 12): Per audit recommendation, not a new milestone. Outcome: Single coherent v1.0 record with extension subsection.

**Lenient error mode on route-level graph cleanup** (Phase 13): Log and continue rather than HTTP 500. Outcome: Matches blob/Qdrant delete semantics; worker safety net is the consistency backstop.

**Always-exposed response fields** (Phase 14): Graph fields always present on DocumentResponse regardless of gate state. Outcome: Simpler API contract; no FastAPI conditional serialization.

**Id-keyed lookup via vertex name attribute** (Phase 15): Robust against entity list reordering. Outcome: Removes hidden "list index == vertex index" invariant.
