# Milestone v1.0: MVP

**Status:** ✅ SHIPPED 2026-03-04
**Phases:** 1-7
**Total Plans:** 8

## Overview

DocIngest goes from zero to a working multi-tenant document ingestion engine in 7 phases. We start by getting the local dev environment running (Docker Compose, no cloud deps), then build the pipeline layer by layer: parsing documents, chunking and embedding them, enabling search, layering on tenant-scoped auth, and finally hardening with proper error handling, status tracking, and observability. Phase 7 cleans up tech debt identified during the milestone audit.

## Phases

### Phase 1: Foundation & Infrastructure

**Goal**: Local dev environment runs all services without cloud dependencies
**Depends on**: Nothing (first phase)
**Plans**: 1 plan

- [x] 01-01: Replace Azure with Local Alternatives

## Milestone Summary

**Key Decisions:**
- All Azure dependencies replaced with local alternatives (MinIO for blob storage, FastEmbed for embeddings)
- Single MinIO bucket with tenant_id path prefix (not per-tenant containers)
- Embedding dimensions changed from 1536 to 384 (bge-small-en-v1.5)
- Rate limiter uses separate Redis connection from ARQ pool
- Rate limiter fails open when Redis unavailable
- Error classification uses plain strings (no enum) for flexibility
- structlog contextvars for trace_id propagation across request and worker boundaries
- Per-stage timing uses time.monotonic() for accurate elapsed measurement

**Issues Resolved:**
- Converter→chunker job name mismatch (pipeline never fired chunking step)
- Async/await precedence bug in search_chunks (AttributeError at runtime)
- Filter AND/OR semantics in Qdrant search (incorrect cross-field filters)
- Health endpoint returning tuple instead of JSONResponse
- Missing document records silently returning instead of failing

**Issues Deferred:**
- None — all v1 requirements satisfied

**Technical Debt Incurred:**
- None remaining — Phase 7 cleaned up all audit-identified tech debt
