<task type="auto">
  <name>Task 1: Create Redis token bucket rate limiter service</name>
  <files>src/docingest/services/rate_limiter.py</files>
  <action>
    Create rate limiter service using Redis token bucket algorithm with atomic Lua script.

    **Redis client:** Use `redis.asyncio.Redis` (available as transitive dependency of ARQ's `redis` package). Parse connection from `settings.redis_url`. Manage connection lifecycle with `init_rate_limiter()` and `close_rate_limiter()` module-level functions (same pattern as db/redis.py).

    **Token bucket Lua script:**
    - Key format: `rate_limit:{key_hash}`
    - Store as Redis hash: `tokens` (float), `last_refill` (float timestamp)
    - On each call: calculate elapsed time since last_refill, add tokens at rate of (limit / 60) per second, cap at limit, decrement by 1 if tokens >= 1
    - Return: [allowed (0/1), remaining (int), reset_seconds (int)]
    - TTL: Set key expiry to 120 seconds (2x the 60s window) so stale keys auto-cleanup

    **Python interface:**
    ```python
    @dataclass
    class RateLimitResult:
        allowed: bool
        limit: int
        remaining: int
        reset: int  # seconds until full bucket

    async def check_rate_limit(key_hash: str, limit: int) -> RateLimitResult:
    ```

    - `limit` comes from the api_key document's `rate_limit` field (requests per minute)
    - If Redis is unavailable, fail open (allow request, log warning) — rate limiting should not block the API when Redis is down
    - Do NOT use the ARQ pool from db/redis.py — create a separate connection for rate limiting to avoid coupling with job queue internals
  </action>

  <done>Rate limiter module exists with Lua-based token bucket, init/close lifecycle, and fail-open behavior</done>

<task type="auto">
  <name>Task 2: Integrate rate limiting into auth flow and add response headers</name>
  <files>src/docingest/api/auth.py, src/docingest/api/middleware.py, src/docingest/api/app.py</files>
  <action>
    **auth.py changes:**
    - Import `check_rate_limit` and `RateLimitResult` from `services.rate_limiter`
    - Add `resolve_tenant_with_rate_limit` dependency that:
      1. Calls existing `resolve_tenant()` to get tenant dict
      2. Calls `check_rate_limit(hash_api_key(api_key), tenant["rate_limit"])`
      3. Stores rate limit result in `request.state.rate_limit` for middleware to read
      4. If not allowed: raise HTTPException(429, detail="Rate limit exceeded") with `Retry-After` header via `headers={"Retry-After": str(result.reset)}`
      5. Returns tenant dict (same shape as before)
    - Update `Tenant` type alias to use `resolve_tenant_with_rate_limit`
    - Keep `resolve_tenant` as-is for internal use (workers don't rate limit)
    - The dependency needs `Request` from fastapi to set `request.state`

    **middleware.py (new file):**
    - Create `RateLimitHeaderMiddleware` as a Starlette `BaseHTTPMiddleware`
    - After response, check `request.state` for `rate_limit` (a `RateLimitResult`)
    - If present, add headers to response:
      - `X-RateLimit-Limit: {result.limit}`
      - `X-RateLimit-Remaining: {result.remaining}`
      - `X-RateLimit-Reset: {result.reset}`
    - If not present (health endpoint, no auth), skip — do not add headers
    - Handle the case where request.state doesn't have rate_limit attribute (use getattr with default None)

    **app.py changes:**
    - Import middleware and rate limiter lifecycle functions
    - Add `app.add_middleware(RateLimitHeaderMiddleware)` after app creation
    - In lifespan startup: call `init_rate_limiter()` to create Redis connection
    - In lifespan shutdown: call `close_rate_limiter()` before other cleanup

    **Error response for 429:**
    - Body: `{"detail": "Rate limit exceeded"}`
    - Headers: `Retry-After: {seconds}`, plus X-RateLimit-* headers (remaining=0)
  </action>
